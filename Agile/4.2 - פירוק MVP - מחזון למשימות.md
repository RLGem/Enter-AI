## 4.2.1 מה זה MVP ומה זה לא — המינימום שמספק ערך

### הבעיה: "בוא נבנה את הכל ואז נראה"

הסיפור חוזר על עצמו: הלקוח רוצה מערכת שלמה. הצוות יוצא ל-6 חודשי פיתוח. בסוף — המערכת לא עונה על הצורך, הדרישות השתנו, התקציב נגמר, ואף אחד לא מרוצה. הבעיה היא לא שהצוות עבד רע — הבעיה היא שניסו לבנות הכל בבת אחת, בלי לבדוק בדרך אם הכיוון נכון.

### מה זה MVP

**MVP — Minimum Viable Product** — הוא הגרסה הקטנה ביותר של המוצר שמספקת ערך אמיתי למשתמשים אמיתיים. לא חצי מוצר. לא פרוטוטייפ. לא הדגמה. אלא **מוצר מינימלי שעובד ושאנשים אמיתיים יכולים להשתמש בו ולקבל ממנו תועלת.**

המילה החשובה ביותר היא **Viable** — בר-קיימא. זה חייב לעבוד, להיות שמיש, ולפתור בעיה אמיתית. המילה **Minimum** אומרת שאנחנו לא מוסיפים שום דבר מעבר למה שנדרש כדי לספק את הערך הזה.

### האיור המפורסם: מסקייטבורד למכונית

הדרך הטובה ביותר להבין MVP היא דרך האיור של Henrik Kniberg. נניח שהלקוח צריך להגיע מנקודה A לנקודה B:

**הגישה הלא נכונה — לבנות מכונית בחלקים:**

```
ספרינט 1       ספרינט 2       ספרינט 3       ספרינט 4       ספרינט 5

  ___             ___            ___             ___          .---.
 /   \           /   \  /   \  /   \ |   | /   \  ___       /|   |\
 \___/           \___/  \___/  \___/ |___| \___/ |   |     | |   | |
                                                 |___|     |_|___|_|
 גלגל            שלדה           שלדה +          שלדה +     מכונית
 בודד           + גלגלים       מנוע            מרכב        שלמה!

 הלקוח:         הלקוח:         הלקוח:          הלקוח:      הלקוח:
 "מה אני        "עדיין לא      "עדיין לא       "עדיין לא   "סוף סוף!
  עושה           יכול           יכול            יכול        אבל רציתי
  עם זה?"        לנסוע"         לנסוע"          לנסוע"      ג'יפ..."
```

**הלקוח לא יכול להשתמש בשום דבר עד הסוף.** כל ספרינט מייצר "חלק" שאי אפשר לעשות איתו כלום. אין פידבק, אין למידה, אין ערך — עד הרגע האחרון.

**הגישה הנכונה — MVP בכל שלב:**

```
ספרינט 1       ספרינט 2       ספרינט 3       ספרינט 4       ספרינט 5

    __            __                            ___          .---.
  _/  \          / |\           /~~\           /|   |\      /|   |\
 |  o  |        /  | \      __/    \__        | |   | |    | |   | |
 /\___/\       /   |  \    /  O    O  \       |_|___|_|    |_|___|_|
O       O     O ===|=  O  O    ____    O

סקייטבורד      קורקינט       אופניים         אופנוע        מכונית

הלקוח:         הלקוח:         הלקוח:          הלקוח:        הלקוח:
"אני יכול      "כבר יותר      "מעולה!         "עכשיו גם    "מושלם,
 לנוע!"         מהר!"          מגיע            מהר           ובדיוק
                               לעבודה!"        וגם בגשם"     מה שצריך"
```

**בכל שלב הלקוח מקבל מוצר שלם שעובד.** סקייטבורד הוא לא "גלגל של מכונית" — הוא אמצעי תחבורה שלם, מינימלי, שמאפשר תנועה. הלקוח משתמש, נותן פידבק, והצוות לומד מה באמת צריך לשפר.

### מה MVP הוא **לא**

| מה אנשים חושבים ש-MVP זה | למה זה לא MVP |
|---------------------------|---------------|
| **חצי מוצר** — מערכת עם פיצ'רים חסרים שלא עובדת | MVP הוא מוצר שלם — רק עם פחות פיצ'רים. כל פיצ'ר שיש בו עובד לגמרי. |
| **פרוטוטייפ** (Prototype) — מודל להדגמה שלא מיועד לשימוש אמיתי | MVP מיועד למשתמשים אמיתיים. פרוטוטייפ מיועד לבדיקת רעיון. |
| **הדגמה** (Demo) — מצגת או סימולציה | MVP הוא תוכנה עובדת עם נתונים אמיתיים, לא הצגה. |
| **גרסה מלאה באגים** — "נשחרר מהר ונתקן אחר כך" | MVP חייב לעמוד ב-Definition of Done. מינימלי לא אומר רשלני. |
| **הגרסה הזולה ביותר שאפשר** — חיתוך פינות | MVP הוא הגרסה **החכמה** ביותר — עם המקסימום ערך במינימום מאמץ. |

### כלל אצבע לזיהוי MVP

שאלו את עצמכם שלוש שאלות:

1. **האם משתמש אמיתי יכול לבצע את הפעולה המרכזית?** (אם לא — זה לא MVP, זה פרוטוטייפ)
2. **האם המשתמש מקבל ערך אמיתי מהשימוש?** (אם לא — זה לא MVP, זה דמו)
3. **האם אפשר להוריד עוד משהו בלי לפגוע בערך המרכזי?** (אם כן — זה עדיין לא מינימלי מספיק)

> **כלל הזהב של MVP:** אם אתה לא קצת מתבייש מהגרסה הראשונה — שחררת מאוחר מדי.
> — Reid Hoffman, מייסד LinkedIn

## 4.2.2 User Story Mapping — טכניקה לפירוק מוצר

### מה זה User Story Mapping

**User Story Mapping** היא טכניקה שהמציא Jeff Patton לפירוק מוצר שלם לחתיכות עבודה ניתנות לניהול. היא עובדת ככה: במקום רשימה שטוחה של Stories (שזה מה שרוב ה-Backlogs נראים), אתה **ממפה את מסע המשתמש** על ציר אופקי, ואז **מפרק כל שלב לפי עדיפות** על ציר אנכי.

### המבנה: ציר אופקי = מסע המשתמש, ציר אנכי = עדיפות

```
  ציר אופקי — מסע המשתמש (User Journey) משמאל לימין
  ─────────────────────────────────────────────────────────────────►

  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ הרשמה    │  │ התחברות  │  │ חיפוש    │  │ הזמנה    │  │ תשלום    │  ← Activities
  │          │  │          │  │ מוצרים   │  │          │  │          │     (פעילויות)
  └──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘
  ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─
  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ הרשמה    │  │ Login    │  │ חיפוש    │  │ הוספה    │  │ תשלום    │
  │ עם מייל  │  │ עם סיסמה │  │ לפי שם   │  │ לסל      │  │ בכרטיס   │ ── MVP
  │ וסיסמה   │  │          │  │          │  │          │  │ אשראי    │   (Release 1)
  ├──────────┤  ├──────────┤  ├──────────┤  ├──────────┤  ├──────────┤
  │ הרשמה    │  │ Login    │  │ סינון    │  │ עדכון    │  │ תשלום    │
  │ עם       │  │ עם       │  │ לפי      │  │ כמויות   │  │ ב-PayPal │ ── Release 2
  │ Google   │  │ Google   │  │ קטגוריה  │  │ בסל      │  │          │
  ├──────────┤  ├──────────┤  ├──────────┤  ├──────────┤  ├──────────┤
  │ הרשמה    │  │ 2FA      │  │ חיפוש    │  │ שמירת    │  │ תשלום    │
  │ עם       │  │          │  │ מתקדם    │  │ מועדפים  │  │ בהעברה   │ ── Release 3
  │ Apple    │  │          │  │ + מיון   │  │          │  │ בנקאית  │
  └──────────┘  └──────────┘  └──────────┘  └──────────┘  └──────────┘

  ▲
  │  ציר אנכי — עדיפות (למעלה = חשוב יותר, למטה = פחות)
  │
```

**כלל חשוב:** קו ה-MVP חוצה את המפה **אופקית**. הוא כולל את הפעולה הפשוטה ביותר מכל שלב במסע, כך שהמשתמש יכול לעבור את כל המסע מתחילתו ועד סופו — גם אם בצורה בסיסית.

### איך לנהל סדנת Story Mapping (1-2 שעות)

#### הכנה (לפני הסדנה)

| פריט | פירוט |
|------|-------|
| **מי מוזמן** | PO (חובה), SM (חובה), מפתחים (חובה), לקוח/בעל עניין (מומלץ מאוד) |
| **חומרים** | Post-its בצבעים שונים, טוש, קיר/לוח גדול, או כלי דיגיטלי (Miro, FigJam) |
| **הכנה** | PO מכין תיאור קצר של המוצר/פיצ'ר, פרסונה אחת לפחות, מטרת המוצר |
| **זמן** | 1-2 שעות (Timebox!) |

#### מהלך הסדנה — צעד אחר צעד

**שלב 1 — הגדרת הפרסונה (10 דקות)**

> "מי המשתמש העיקרי שלנו?"

כתבו את הפרסונה על Post-it בצבע ייחודי ותלו בפינה. לדוגמה: "שרה, מנהלת משאבי אנוש בחברת היי-טק, 35, רוצה לנהל עובדים ביעילות."

**שלב 2 — מיפוי הפעילויות (Activities) — הציר האופקי (15 דקות)**

> "מה שרה עושה עם המערכת, מתחילת האינטראקציה ועד סופה?"

כתבו כל פעילות מרכזית על Post-it (צבע אחד — למשל כחול) וסדרו אותן משמאל לימין לפי סדר כרונולוגי. אלו הן ה-Activities — הכותרות של כל עמודה.

דוגמאות: "הרשמה" → "התחברות" → "הוספת עובד" → "צפייה בדוחות" → "ניהול חופשות"

**שלב 3 — פירוק לפעולות (Tasks) — מתחת לכל Activity (20 דקות)**

> "בתוך כל פעילות — מה הצעדים הקונקרטיים?"

כתבו כל פעולה על Post-it (צבע אחר — למשל צהוב) ותלו מתחת לפעילות הרלוונטית, מלמעלה למטה לפי חשיבות.

דוגמה תחת "הוספת עובד": "מילוי פרטים בסיסיים" → "הוספת תמונה" → "שיוך למחלקה" → "הגדרת הרשאות"

**שלב 4 — ציור קו ה-MVP (15 דקות)**

> "מה המינימום שמספיק כדי ששרה תוכל לבצע את כל המסע?"

ציירו קו אופקי. כל מה שמעל הקו = MVP. הכלל: **בכל עמודה חייב להיות לפחות Post-it אחד מעל הקו.** אחרת — המסע קטוע והמשתמש תקוע.

**שלב 5 — הגדרת Releases (15 דקות)**

> "אחרי ה-MVP, מה בא ב-Release הבא? ואחריו?"

ציירו קווים נוספים מתחת לקו ה-MVP. כל "שכבה" היא Release. סמנו כל Release בשם ותאריך משוער.

**שלב 6 — תיקוף וסיכום (15 דקות)**

> "בואו נעבור על המפה מתחילתה ועד סופה ונבדוק הגיון."

קראו את המפה בקול כסיפור: "שרה נרשמת עם מייל, מתחברת, מוסיפה עובד עם פרטים בסיסיים, רואה דוח פשוט, ומנהלת חופשות ידנית." אם הסיפור לא הגיוני — תקנו.

#### תוצר הסדנה

בסוף הסדנה יש לכם:

- **מפה ויזואלית** של כל המוצר
- **קו MVP** ברור — מה בגרסה הראשונה
- **Releases** עתידיים מזוהים
- **בסיס** ליצירת Epics, Features ו-Stories ב-Azure DevOps

> **טיפ:** צלמו את הלוח/הקיר. אל תסמכו על הזיכרון. העבירו את התוצר ל-Azure DevOps באותו שבוע, לפני שה-Post-its נופלים.

## 4.2.3 MoSCoW — Must / Should / Could / Won't

### מה זה MoSCoW

**MoSCoW** היא שיטת תעדוף שעוזרת לקבוע מה חייב להיות ב-MVP ומה לא. השם הוא ראשי תיבות:

| אות | משמעות | הסבר | כלל אצבע |
|-----|--------|------|-----------|
| **Mo** | **Must Have** | חייבים. בלי זה המוצר לא עובד. אם זה חסר — אי אפשר לשחרר. | ~60% מהמאמץ |
| **S** | **Should Have** | צריכים. חשוב מאוד, אבל אפשר לחיות בלי זה בגרסה הראשונה. פתרון עוקף קיים. | ~20% מהמאמץ |
| **Co** | **Could Have** | אפשר. נחמד שיהיה (Nice to Have). אם נשאר זמן — נכניס. | ~20% מהמאמץ |
| **W** | **Won't Have (this time)** | לא הפעם. מוגדר במפורש שלא ייכנס לגרסה הנוכחית. לא "לא" — אלא "לא עכשיו". | 0% |

### למה MoSCoW עובד

הבעיה המרכזית בתעדוף היא ש**הכל חשוב**. כשמבקשים מלקוח לתעדף, הוא שם הכל ב-"חובה". MoSCoW מכריח לחשוב בצורה יחסית: אם 60% מהמאמץ הולך ל-Must — מה באמת **חייב** להיות שם?

### דוגמה מעשית: מערכת ניהול עובדים

נניח שאנחנו בונים מערכת HR פנימית. הלקוח רוצה 10 פיצ'רים. בואו נעביר אותם דרך MoSCoW:

**רשימת הפיצ'רים:**

| # | פיצ'ר | תיאור קצר |
|:-:|-------|-----------|
| 1 | הרשמת עובד | הוספת עובד חדש עם פרטים בסיסיים |
| 2 | צפייה ברשימת עובדים | טבלה עם כל העובדים |
| 3 | עריכת פרטי עובד | שינוי פרטים של עובד קיים |
| 4 | ניהול חופשות | בקשות חופשה ואישורן |
| 5 | דוחות מתקדמים | גרפים, ייצוא ל-Excel, פילטרים |
| 6 | אימות דו-שלבי | 2FA בכניסה למערכת |
| 7 | ייבוא עובדים מ-Excel | טעינה מרוכזת של עובדים |
| 8 | התראות מייל | התראות אוטומטיות על אירועים |
| 9 | ממשק בערבית | תמיכה בשפה נוספת |
| 10 | אפליקציית מובייל | גרסה סלולרית |

**תהליך ההחלטה:**

לגבי כל פיצ'ר, שואלים את השאלות הבאות:

> **שאלה 1:** "האם המערכת חסרת תועלת בלי הפיצ'ר הזה?"
> אם כן → **Must Have**
>
> **שאלה 2:** "האם יש פתרון עוקף סביר (Workaround)?"
> אם אין פתרון עוקף → **Must Have**
> אם יש פתרון עוקף אבל הוא מציק → **Should Have**
>
> **שאלה 3:** "האם אפשר לחכות Release אחד?"
> אם כן בלי בעיה → **Could Have**
>
> **שאלה 4:** "האם זה לא רלוונטי עכשיו בכלל?"
> אם כן → **Won't Have**

**התוצאה:**

| פיצ'ר | MoSCoW | נימוק |
|-------|:------:|-------|
| הרשמת עובד | **Must** | בלי זה אין מערכת. אי אפשר לעשות כלום. |
| צפייה ברשימת עובדים | **Must** | בלי זה אין ערך — הכנסנו עובדים ולא רואים אותם? |
| עריכת פרטי עובד | **Must** | טעויות יקרו. בלי יכולת עריכה — המערכת תהפוך ללא שמישה. |
| ניהול חופשות | **Should** | חשוב מאוד, אבל בגרסה ראשונה — אפשר לנהל חופשות במייל/Excel. פתרון עוקף קיים. |
| דוחות מתקדמים | **Could** | נחמד, אבל אפשר לייצא נתונים ידנית. לא חוסם. |
| אימות דו-שלבי | **Should** | אבטחה חשובה, אבל למערכת פנימית ברשת הארגונית — אפשר בגרסה הבאה. |
| ייבוא עובדים מ-Excel | **Could** | אם יש 500 עובדים — זה חשוב. אם 30 — אפשר להקליד ידנית. תלוי הקשר. |
| התראות מייל | **Could** | נחמד, אבל לא חוסם. אפשר לבדוק את המערכת ידנית. |
| ממשק בערבית | **Won't** | לא הפעם. 90% מהמשתמשים דוברי עברית. נטפל בזה ב-Release עתידי. |
| אפליקציית מובייל | **Won't** | לא הפעם. המערכת מותאמת דפדפן, ורוב העובדים עובדים ממחשב. |

**סיכום:**

```
Must Have (60%):     הרשמה, צפייה, עריכה          ← חייבים ב-MVP
Should Have (20%):   חופשות, 2FA                   ← Release 2
Could Have (20%):    דוחות, ייבוא, התראות          ← Release 3 (אם נשאר זמן)
Won't Have:          ערבית, מובייל                  ← לא בתכנון הנוכחי
```

### טעויות נפוצות ב-MoSCoW

| טעות | מה קורה | איך למנוע |
|------|---------|-----------|
| **הכל Must** | אין תעדוף אמיתי, הצוות טובע | הגבילו Must ל-60% מהמאמץ הכולל. אם 80% הם Must — תעדפו שוב. |
| **Won't = לא נעשה לעולם** | הלקוח נבהל | הדגישו: Won't = **לא הפעם**. זה נכנס ל-Backlog לעתיד. |
| **אין Won't** | לא לקחנו החלטות קשות | אם אין Won't — לא באמת תעדפנו. תמיד צריך לוותר על משהו. |
| **תעדוף ללא הלקוח** | הלקוח לא מסכים ויש חיכוכים | הלקוח חייב להשתתף בתעדוף MoSCoW, או לפחות לאשר את התוצאה. |

## 4.2.4 פירוק: Vision → Epics → Features → Stories

### ההיררכיה המלאה

כל מוצר תוכנה, מהחזון הגדול ביותר ועד המשימה הקטנה ביותר, מפורק להיררכיה ברורה. האיור הבא מציג את הרמות ואת היחס ביניהן:

```
Vision (חזון המוצר)
│
│   "מערכת HR שמנהלת את כל מחזור חיי העובד בארגון"
│
├── Epic 1: ניהול עובדים
│   │
│   ├── Feature 1.1: הרשמת עובד חדש
│   │   ├── Story: בתור מנהל HR, אני רוצה להוסיף עובד עם פרטים בסיסיים
│   │   │   ├── Task: עיצוב טופס הוספת עובד
│   │   │   ├── Task: פיתוח API להוספת עובד
│   │   │   ├── Task: פיתוח ממשק משתמש — טופס הוספה
│   │   │   ├── Task: כתיבת Unit Tests
│   │   │   └── Task: כתיבת Integration Tests
│   │   ├── Story: בתור מנהל HR, אני רוצה לקבל וידוא שלא קיים עובד כפול
│   │   └── Story: בתור מנהל HR, אני רוצה לקבל אישור הצלחה אחרי הוספה
│   │
│   ├── Feature 1.2: צפייה ברשימת עובדים
│   │   ├── Story: בתור מנהל HR, אני רוצה לראות טבלה של כל העובדים
│   │   ├── Story: בתור מנהל HR, אני רוצה לחפש עובד לפי שם
│   │   └── Story: בתור מנהל HR, אני רוצה לסנן עובדים לפי מחלקה
│   │
│   └── Feature 1.3: עריכת פרטי עובד
│       ├── Story: בתור מנהל HR, אני רוצה לערוך פרטים בסיסיים של עובד
│       └── Story: בתור מנהל HR, אני רוצה לראות היסטוריית שינויים
│
├── Epic 2: ניהול חופשות
│   │
│   ├── Feature 2.1: הגשת בקשת חופשה
│   ├── Feature 2.2: אישור/דחיית חופשה
│   └── Feature 2.3: דוח יתרת חופשה
│
├── Epic 3: דוחות ואנליטיקה
│   └── ...
│
└── Epic 4: הרשאות ואבטחה
    └── ...
```

### הגדרת כל רמה

| רמה | הגדרה | גודל | מי אחראי | דוגמה |
|-----|--------|------|----------|-------|
| **Vision** | מטרת המוצר הכוללת. משפט אחד שמסביר **למה** המוצר קיים. | הפרויקט כולו | PO + הנהלה | "מערכת HR שמפשטת ניהול עובדים ומחליפה Excel" |
| **Epic** | אוסף גדול של עבודה שקשור לנושא אחד. חוצה מספר ספרינטים. | **מספר ספרינטים** (חודשים) | PO | "ניהול עובדים" |
| **Feature** | יכולת ספציפית בתוך Epic. ניתנת לפיתוח ב-1-2 ספרינטים. | **1-2 ספרינטים** (3-6 שבועות) | PO | "הרשמת עובד חדש" |
| **User Story** | פיסת עבודה שמספקת ערך למשתמש. נכנסת בספרינט אחד. | **נכנס בספרינט אחד** (ימים-שבוע) | PO כותב, צוות מעריך | "בתור מנהל HR, רוצה להוסיף עובד עם פרטים בסיסיים" |
| **Task** | פעולה טכנית קונקרטית שמפתח מבצע. | **שעות** (2-16 שעות) | מפתח | "פיתוח API endpoint: POST /employees" |

### כללי אצבע: מתי להשתמש בכל רמה

```
שאלה: "על מה מדברים?"                    → רמה:

"הפרויקט כולו, החזון, הכיוון"            → Vision
"נושא גדול שלוקח חודשים"                 → Epic
"יכולת מוגדרת שלוקחת 1-2 ספרינטים"      → Feature
"דבר אחד שמשתמש יכול לעשות"              → Story
"דבר אחד שמפתח צריך לעשות"               → Task
```

### כללי אצבע: מתי לפרק לרמה הבאה

| אם... | אז... |
|--------|--------|
| Story לא נכנס בספרינט אחד | פצלו ל-2 Stories או יותר |
| Feature לוקח יותר מ-2 ספרינטים | שקלו לפצל ל-2 Features |
| Task לוקח יותר מ-16 שעות | פצלו ל-Tasks קטנים יותר |
| Epic לא ברור מה כולל | הגדירו את ה-Features שבו |
| לא יודעים לתת Story Points | כנראה שה-Story גדול מדי או לא ברור — צריך Refinement |

### היחס בין הרמות ב-Azure DevOps

```
Azure DevOps          ↔    Agile Term
────────────────────────────────────────
Epic                  ↔    Epic
  └── Feature         ↔    Feature
        └── PBI *     ↔    User Story
              └── Task↔    Task

* PBI = Product Backlog Item (בתהליך Scrum ב-Azure DevOps)
  או User Story (בתהליך Agile ב-Azure DevOps)
```

## 4.2.5 Release Planning — תכנון גרסאות מעבר לספרינט בודד

### למה צריך Release Planning

Sprint Planning מטפל ב-"מה נעשה ב-3 השבועות הקרובים." אבל בעלי עניין שואלים שאלות אחרות:

- "מתי הגרסה הראשונה תהיה מוכנה?"
- "מתי פיצ'ר X יהיה זמין?"
- "מה יהיה מוכן עד סוף הרבעון?"

**Release Planning** עונה על השאלות האלה. הוא מסתכל על **כמה ספרינטים קדימה** ומתכנן **מתי כל Feature/Epic צפוי להיות מוכן.**

### Release לא חייב להיות Deploy

חשוב להבין: **Release הוא נקודת זמן שבה אוסף של Features מוכן למסירה.** זה לא בהכרח אומר Deploy ל-Production. אפשר לעשות Release לסביבת Staging, לקבוצת משתמשים מצומצמת (Beta), או לסביבת הלקוח.

### איך לבנות תוכנית Release

#### שלב 1: הגדרת ה-Releases

הגדירו כמה Releases מתוכננים ומה המטרה של כל אחד:

| Release | מטרה | תאריך יעד משוער |
|---------|------|-----------------|
| **Release 1 (MVP)** | מערכת HR בסיסית — הוספה, צפייה, עריכה של עובדים | סוף ספרינט 3 |
| **Release 2** | ניהול חופשות + 2FA | סוף ספרינט 6 |
| **Release 3** | דוחות מתקדמים + ייבוא + התראות | סוף ספרינט 9 |

#### שלב 2: שיוך Features ל-Releases

מתוך ה-MoSCoW (סעיף 4.2.3) ומתוך ה-Story Map (סעיף 4.2.2), שייכו כל Feature ל-Release:

```
Release 1 (MVP) — ספרינטים 1-3
──────────────────────────────
Sprint 1:  Feature 1.1 — הרשמת עובד חדש
Sprint 2:  Feature 1.2 — צפייה ברשימת עובדים
Sprint 3:  Feature 1.3 — עריכת פרטי עובד
           + תיקון באגים, QA, הכנה לשחרור

Release 2 — ספרינטים 4-6
──────────────────────────────
Sprint 4:  Feature 2.1 — הגשת בקשת חופשה
Sprint 5:  Feature 2.2 — אישור/דחיית חופשה
           Feature 4.1 — 2FA
Sprint 6:  Feature 2.3 — דוח יתרת חופשה
           + תיקון באגים, QA, הכנה לשחרור

Release 3 — ספרינטים 7-9
──────────────────────────────
Sprint 7:  Feature 3.1 — דוחות בסיסיים
Sprint 8:  Feature 3.2 — ייבוא מ-Excel
Sprint 9:  Feature 3.3 — התראות מייל
           + תיקון באגים, QA, הכנה לשחרור
```

#### שלב 3: חישוב אם התוכנית ריאלית

**השאלה המרכזית:** "האם יש לנו מספיק Velocity כדי לסיים את כל ה-Stories בזמן?"

| נתון | ערך | הסבר |
|------|-----|------|
| Velocity ממוצע | 30 Story Points לספרינט | ממוצע 3 הספרינטים האחרונים |
| Release 1 — סה"כ הערכה | 85 Story Points | סכום כל ה-Stories ב-Features 1.1-1.3 |
| ספרינטים זמינים | 3 | ספרינטים 1-3 |
| Capacity | 3 x 30 = 90 Points | קיבולת תיאורטית |
| **מסקנה** | 85 < 90 — **ריאלי, עם מרווח קטן** | |

> **כלל אצבע:** תמיד השאירו **מרווח של 15-20%** מה-Capacity לבאגים, חסמים, ועבודה לא מתוכננת. אם ה-Velocity שלכם 30, תכננו על 25 Points בפועל.

#### שלב 4: תקשורת עם בעלי עניין

**Release Plan הוא כלי תקשורת, לא חוזה.** הציגו אותו כך:

```
"על בסיס ה-Velocity שלנו (30 Points לספרינט),
אנחנו מעריכים שה-MVP יהיה מוכן בסוף ספרינט 3
(בערך [תאריך]).

זו הערכה, לא התחייבות.
ככל שנתקדם ונלמד — נעדכן."
```

### טיפים ל-Release Planning

| טיפ | הסבר |
|-----|------|
| **תכננו בטווחים, לא בתאריכים מדויקים** | "סוף מרץ — אמצע אפריל" עדיף על "31 במרץ" |
| **עדכנו כל ספרינט** | אחרי כל Sprint Review — בדקו אם התוכנית עדיין ריאלית |
| **הציגו אי-ודאות** | השתמשו ב-"Cone of Uncertainty" — ככל שהתאריך רחוק יותר, הטווח רחב יותר |
| **אל תתחייבו על תאריך לפני שיש Velocity** | בספרינט הראשון אין Velocity. אל תבטיחו תאריכים. |
| **שמרו על Buffer** | לפחות ספרינט אחד "רזרבי" לפני Release גדול — לבאגים, QA, תיקונים |

### מה קורה כשהתוכנית לא עובדת?

כשהמציאות לא תואמת את התוכנית (וזה **יקרה**), יש ארבע אפשרויות:

| אפשרות | מתי מתאימה |
|---------|-----------|
| **הורדת Scope** — הוצאת Features מה-Release | כשהתאריך קריטי |
| **דחיית תאריך** — הזזת ה-Release | כשה-Scope קריטי |
| **הוספת משאבים** — הגדלת הצוות (בזהירות!) | רק אם יש זמן לקליטה. אחרת — יאט במקום לזרז (חוק ברוקס) |
| **הקטנת Stories** — פישוט Features קיימים | הדרך הטובה ביותר — לעשות פחות, אבל טוב |

> **חוק ברוקס:** "הוספת כוח אדם לפרויקט תוכנה מאוחר — מאחרת אותו יותר." מפתחים חדשים צריכים חניכה, הצוות מאט כדי להסביר, ותקשורת מסתבכת.

## 4.2.6 דוגמה מלאה — פירוק פיצ'ר אמיתי מ-MVP עד Tasks

### הקונטקסט

**מוצר:** מערכת ניהול משתמשים (User Management System) לאפליקציית SaaS פנימית.

**חזון (Vision):**
> "מערכת שמאפשרת למנהלי מערכת לנהל משתמשים, הרשאות, ותפקידים — כך שכל משתמש רואה רק את מה שרלוונטי לו."

**פרסונה:**
> "דנה, מנהלת מערכת (System Admin), 32, אחראית על 200 משתמשים באפליקציה. רוצה להוסיף/להסיר משתמשים, לנהל הרשאות, ולראות מי מחובר."

```
Vision: מערכת ניהול משתמשים
│
├── Epic 1: ניהול משתמשים (CRUD)           ← Must
├── Epic 2: ניהול הרשאות ותפקידים          ← Must

Epic 1: ניהול משתמשים (CRUD)

### שלב 3: פירוק Feature 1.1 ל-Stories

```
Feature 1.1: יצירת משתמש חדש

│

│           הערכה: 6 שעות

├── Task 5: כתיבת Unit Tests — Backend
│           הערכה: 3 שעות
│           מבצע: מפתח Backend
│           תיאור: Tests ל-API:
│                   - יצירה מוצלחת

## 4.2.7 Azure DevOps — Epics, Features, Delivery Plans

### יצירת ההיררכיה ב-Azure DevOps

#### יצירת Epic

1. לכו ל-**Boards** → **Backlogs**
2. בחלק העליון, ודאו שאתם בתצוגת **Epics** (בחרו מה-Dropdown למעלה)
3. לחצו **+ New Work Item**
4. מלאו:
   - **Title:** "ניהול משתמשים (CRUD)"
   - **Description:** תיאור מפורט של ה-Epic
   - **Area Path:** הפרויקט/הצוות
   - **Iteration Path:** ריק (Epic חוצה ספרינטים)
   - **Priority:** 1 (גבוה)
   - **State:** New

```
Boards → Backlogs → Epics (dropdown) → + New Work Item
                                          │
                                          ▼
                                    Title: ניהול משתמשים (CRUD)
                                    State: New
                                    Priority: 1
```

#### יצירת Feature תחת Epic

1. פתחו את ה-Epic שיצרתם
2. בלשונית **Related Work** (או **Child Items**), לחצו **+ Add link** → **New item**
3. בחרו סוג: **Feature**
4. מלאו:
   - **Title:** "יצירת משתמש חדש"
   - **Iteration Path:** Sprint 1 (אם ה-Feature מתוכנן לספרינט ספציפי)
   - **Priority:** 1
   - **Target Date:** (אופציונלי — תאריך יעד)

**או בדרך מהירה:**

1. לכו ל-**Backlogs** → בחרו **Features** מה-Dropdown
2. לחצו **+ New Work Item**
3. מלאו את הפרטים
4. גררו את ה-Feature מתחת ל-Epic הרלוונטי (Drag & Drop בתצוגת Backlogs)

#### יצירת User Story (PBI) תחת Feature

1. פתחו את ה-Feature
2. **+ Add link** → **New item** → סוג: **Product Backlog Item** (או **User Story**)
3. מלאו:
   - **Title:** "בתור מנהל מערכת, אני רוצה ליצור משתמש חדש עם פרטים בסיסיים"
   - **Description:** תיאור + Acceptance Criteria
   - **Story Points:** 5
   - **Iteration Path:** Sprint 1
   - **State:** New

#### יצירת Task תחת Story

1. פתחו את ה-Story
2. בלשונית **Tasks**, לחצו **+ Add Task**
3. מלאו:
   - **Title:** "פיתוח API endpoint: POST /api/users"
   - **Remaining Work:** 6 (שעות)
   - **Activity:** Development
   - **Assigned To:** שם המפתח

### צפייה בהיררכיה — Portfolio Backlogs

**Portfolio Backlogs** מאפשרים לראות את ההיררכיה המלאה: Epics → Features → Stories.

1. לכו ל-**Boards** → **Backlogs**
2. בחרו **Epics** מה-Dropdown למעלה
3. לחצו על **הרחבה** (▶) ליד כל Epic כדי לראות את ה-Features שמתחתיו
4. לחצו על **הרחבה** ליד כל Feature כדי לראות את ה-Stories

```
Boards → Backlogs → Epics
│
├── ▼ Epic: ניהול משתמשים (CRUD)           Priority: 1
│   ├── ▼ Feature: יצירת משתמש חדש          Sprint 1    State: Active
│   │   ├── Story: יצירה בסיסית             5 SP        State: Active
│   │   ├── Story: מניעת כפילויות           2 SP        State: New
│   │   └── Story: מייל הזמנה               3 SP        State: New
│   │
│   ├── ▼ Feature: רשימת משתמשים            Sprint 1    State: New
│   │   ├── Story: טבלת משתמשים             5 SP        State: New
│   │   └── ...
│   │
│   └── ► Feature: עריכת משתמש              Sprint 2    State: New
│
├── ► Epic: ניהול הרשאות ותפקידים           Priority: 1
└── ► Epic: מעקב ואודיט                     Priority: 2
```

### Delivery Plans — תכנון חוצה ספרינטים

**Delivery Plans** ב-Azure DevOps מאפשר לראות את התוכנית על פני מספר ספרינטים — בדיוק מה שצריך ל-Release Planning.

#### הגדרת Delivery Plan

1. לכו ל-**Boards** → **Delivery Plans**
2. לחצו **+ New Plan**
3. הגדירו:
   - **Name:** "Release Plan — מערכת ניהול משתמשים"
   - **Teams:** בחרו את הצוות שלכם (אפשר יותר מצוות אחד)
   - **Backlog level:** בחרו **Features** (כדי לראות ברמת Feature)
4. לחצו **Create**

#### מה רואים ב-Delivery Plan

```
┌─────────────────────────────────────────────────────────────────────┐
│  Delivery Plan: Release Plan — מערכת ניהול משתמשים                  │
├──────────┬──────────┬──────────┬──────────┬──────────┬──────────────┤
│ Sprint 1 │ Sprint 2 │ Sprint 3 │ Sprint 4 │ Sprint 5 │ Sprint 6    │
│ 1-21 Mar │ 22M-11A  │ 12A-2May │ 3-23 May │ 24M-13J  │ 14J-4Jul   │
├──────────┼──────────┼──────────┼──────────┼──────────┼──────────────┤
│Feature:  │Feature:  │Feature:  │Feature:  │Feature:  │Feature:     │
│יצירת     │עריכת     │שיוך      │Audit Log │דוח       │ייבוא CSV    │
│משתמש     │משתמש     │הרשאות    │          │פעילות    │             │
│          │          │לתפקיד   │          │          │             │
│Feature:  │Feature:  │Feature:  │          │          │             │
│רשימת     │השבתת     │בדיקת     │          │          │             │
│משתמשים   │משתמש     │הרשאות    │          │          │             │
│          │          │          │          │          │             │
│          │Feature:  │          │          │          │             │
│          │הגדרת     │          │          │          │             │
│          │תפקידים   │          │          │          │             │
├──────────┴──────────┴──────────┼──────────┴──────────┴──────────────┤
│   ← Release 1 (MVP) →         │   ← Release 2 →                   │
└────────────────────────────────┴───────────────────────────────────┘
```

#### הוספת Milestones (אבני דרך)

1. בתוך ה-Delivery Plan, לחצו על **Settings** (גלגל שיניים)
2. בחרו **Markers**
3. הוסיפו:
   - **"Release 1 (MVP)"** — תאריך: סוף Sprint 3
   - **"Release 2"** — תאריך: סוף Sprint 6
4. Milestones יופיעו כקו אנכי על ציר הזמן

#### טיפים ל-Delivery Plans

| טיפ | הסבר |
|-----|------|
| **עדכנו Iteration Path** | ודאו שלכל Feature יש Iteration Path נכון — אחרת הוא לא יופיע |
| **השתמשו ב-Filters** | סננו לפי Tags (למשל: "MVP", "Release 2") לראות רק את מה שרלוונטי |
| **שתפו את הלינק** | שלחו את הקישור ל-Delivery Plan לבעלי עניין — הם יכולים לצפות בלי לערוך |
| **הציגו ב-Sprint Review** | הציגו את ה-Delivery Plan בתחילת כל Sprint Review כדי לתת הקשר |

### Query מועיל: "מה ב-Release 1?"

צרו Query שמציג את כל ה-Work Items שקשורים ל-Release 1:

```
Work Item Type IN (Feature, Product Backlog Item, Bug)
AND Iteration Path UNDER @Project\Sprint 1
    OR Iteration Path UNDER @Project\Sprint 2
    OR Iteration Path UNDER @Project\Sprint 3
AND State <> Removed

ORDER BY [Backlog Priority] ASC
```

שמרו את ה-Query בשם **"Release 1 — All Items"** והצמידו ל-Dashboard.

### Dashboard Widget מומלץ

הוסיפו ל-Dashboard של הצוות:

| Widget | מה מציג | למה חשוב |
|--------|---------|----------|
| **Query Results** | כל ה-Items של ה-Release הנוכחי | תמונת מצב |
| **Chart for Work Items** | התפלגות States (New/Active/Closed) לפי Feature | התקדמות |
| **Burndown** (Sprint) | Burndown של הספרינט הנוכחי | מעקב יומי |
| **Cumulative Flow** | זרימה מצטברת לאורך ספרינטים | מגמות |
| **Velocity** | Velocity לאורך הספרינטים האחרונים | תכנון עתידי |
