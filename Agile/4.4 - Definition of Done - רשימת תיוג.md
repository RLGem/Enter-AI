## 4.4.1 למה צריך DoD — "סיימתי" זה לא "עובד"

### הבעיה: כל אחד מתכוון למשהו אחר

יום רביעי, 16:30. ה-Daily Standup. יוסי המפתח אומר בגאווה: **"סיימתי את Story #210."**

מה יוסי מתכוון?

> "כתבתי את הקוד. הוא עובר compilation. עשיתי בדיקה מהירה על המחשב שלי. הכל טוב."

מיכל, ה-PO, שומעת "סיימתי" ומבינה:

> "מצוין! אז זה בסביבת Staging, עבר בדיקות, ואני יכולה להדגים את זה ללקוח ב-Review ביום חמישי."

הלקוח שומע ב-Review שהפיצ'ר "הושלם" ומבין:

> "מעולה, אז המשתמשים שלי כבר יכולים להשתמש בזה."

**ואז מגיע יום חמישי.**

מיכל נכנסת ל-Staging להכין את ה-Demo. הפיצ'ר לא שם. "יוסי, אמרת שסיימת?" — "כן, סיימתי לכתוב את הקוד. לא עשיתי Deploy." — "ואין Unit Tests?" — "אממ... לא הספקתי." — "ו-Code Review?" — "..."

**זה קורה בכל צוות פיתוח בעולם.** לא בגלל שאנשים משקרים — בגלל שלמילה "סיימתי" אין הגדרה אחת.

### הפתרון: Definition of Done

‏**Definition of Done (DoD)** היא רשימת תיוג מוסכמת שמגדירה מה חייב לקרות כדי שמשהו ייחשב "גמור". לא "גמור מבחינתי" — **גמור באמת.**

<div dir="rtl">

| עם DoD | בלי DoD |
|--------:|---------:|
| "סיימתי" = הקוד כתוב, נבדק, עבר Code Review, נפרס, ואושר | "סיימתי" = כתבתי קוד |
| הגדרה אחת לכולם — ללא עמימות | כל מפתח מגדיר "גמור" אחרת |
| באגים מתגלים בזמן הפיתוח | באגים מתגלים ב-Review |
| Increment שאפשר להדגים וגם לשלוח ל-Production | Increment לא אמין |

</div>


> **כלל הזהב:** אם Item לא עומד בכל סעיפי ה-DoD — הוא **לא Done**. לא "כמעט Done", לא "Done חוץ מ...". פשוט לא Done.

## 4.4.2 תבנית DoD — רשימת תיוג מוכנה להתאמה

הרשימות הבאות מוכנות לשימוש. העתיקו, התאימו לצוות שלכם, והשתמשו מהיום.

### ‏DoD ברמת Task

<div dir="rtl">

```
‏Definition of Done — Task
──────────────────────────
- [ ] קוד נכתב ועובר Compilation ללא שגיאות
‏- [ ] Unit Tests כתובים ועוברים (כיסוי לפחות לתרחישים העיקריים)
- [ ] קוד עבר Code Review ואושר על ידי לפחות מפתח אחד נוסף
- [ ] תיעוד עודכן (אם רלוונטי — README, API docs, הערות בקוד)
- [ ] הקוד מוכן ל-Merge — אין Conflicts עם ה-Branch הראשי
```

</div>


### ‏DoD ברמת Story (User Story / PBI)

<div dir="rtl">

```
‏Definition of Done — Story
──────────────────────────
- [ ] כל ה-Tasks של ה-Story הושלמו (עומדים ב-Task DoD)
‏- [ ] Acceptance Criteria — כל הקריטריונים מאומתים ועוברים
- [ ] בדיקות אינטגרציה (Integration Tests) עוברות
- [ ] הקוד נפרס לסביבת Dev/Staging בהצלחה
‏- [ ] PO אישר את הפיצ'ר בסביבת הבדיקות
- [ ] אין באגים פתוחים בחומרה גבוהה (Critical / High) הקשורים ל-Story
```

</div>


### ‏DoD ברמת Sprint

<div dir="rtl">

```
‏Definition of Done — Sprint
──────────────────────────
- [ ] כל ה-Stories שהוכנסו לספרינט עומדים ב-Story DoD
‏- [ ] Sprint Review בוצע — הודגמה תוכנה עובדת
‏- [ ] Sprint Retrospective בוצע — Action Items מתועדים
‏- [ ] Burndown Chart מעודכן ומשקף את המצב הסופי
- [ ] אין Work Items תקועים בסטטוס Active (הכל Done או חזר ל-Backlog)
```

</div>


### ‏DoD ברמת Release

<div dir="rtl">

```
‏Definition of Done — Release
──────────────────────────
- [ ] כל ה-Stories שמיועדים ל-Release עברו QA בסביבת Staging
- [ ] בדיקות רגרסיה (Regression Tests) עברו בהצלחה
‏- [ ] Performance Testing בוצע (אם רלוונטי — לפי מדדים מוגדרים מראש)
‏- [ ] Release Notes מוכנים — רשימת שינויים ברורה
- [ ] תוכנית Rollback קיימת ומתועדת — מה עושים אם משהו נשבר
- [ ] אישור PO לגרסה
```

</div>


> **חשוב:** העתיקו את הרשימות האלה. התאימו לצוות שלכם. הסירו מה שלא רלוונטי, הוסיפו מה שחסר. אבל **אל תתחילו עם רשימה ריקה** — תתחילו עם זה.

## ‏4.4.3 DoD לפי רמה — Task / Story / Sprint / Release

### הרמות בונות אחת על השנייה

כל רמה כוללת את הרמה שמתחתיה. Sprint לא יכול להיות Done אם ה-Stories לא Done. Story לא יכול להיות Done אם ה-Tasks לא Done.

<div dir="rtl">

```
                    ┌─────────────────────┐
                    │                     │
‏                    │      Release        │  ← כל ה-Stories עברו QA,
                    │                     │    רגרסיה, Rollback מוכן
                    │  ┌───────────────┐  │
                    │  │               │  │
‏                    │  │    Sprint     │  │  ← כל ה-Stories Done,
‏                    │  │               │  │    Review + Retro בוצעו
                    │  │  ┌─────────┐  │  │
                    │  │  │         │  │  │
‏                    │  │  │  Story  │  │  │  ← כל ה-Tasks Done,
‏                    │  │  │         │  │  │    AC מאומתים, PO אישר
                    │  │  │ ┌─────┐ │  │  │
‏                    │  │  │ │Task │ │  │  │  ← קוד, Tests, Review,
                    │  │  │ └─────┘ │  │  │    תיעוד
                    │  │  └─────────┘  │  │
                    │  └───────────────┘  │
                    └─────────────────────┘
```

</div>


### מתי בודקים כל רמה

<div dir="rtl">

| מתי | מי בודק | רמה |
|-----:|---------:|-----:|
| כשמעבירים Task ל-Done | המפתח + הסוקר (Code Review) | **Task** |
| כשכל ה-Tasks של ה-Story הושלמו | PO + המפתח | **Story** |
| ביום האחרון, לפני ה-Sprint Review | SM + הצוות | **Sprint** |
| לפני ה-Deploy ל-Production | PO + SM + QA | **Release** |

</div>


### למה 4 רמות ולא אחת?

כי כל רמה עונה על שאלה אחרת:

<div dir="rtl">

| השאלה שהיא עונה עליה | רמה |
|----------------------:|-----:|
| "האם הקוד הזה ראוי?" | **Task** |
| "האם הפיצ'ר הזה עובד ומאושר?" | **Story** |
| "האם ה-Increment הזה שלם ואפשר להדגים אותו?" | **Sprint** |
| "האם הגרסה הזו מוכנה למשתמשים אמיתיים?" | **Release** |

</div>

## 4.4.4 איך לעדכן DoD — תהליך שיפור מתמיד

### ‏DoD הוא מסמך חי

ה-DoD לא נכתב פעם אחת ונשכח. הוא **מתפתח** עם הצוות. כשהצוות מתבגר, ה-DoD הופך מחמיר יותר. כשנגלים בעיות איכות — מוסיפים סעיפים. כשסעיף הופך לאוטומטי וברור מאליו — אפשר (בזהירות) להסיר.

### מתי מעדכנים

<div dir="rtl">

| מה עושים | מתי |
|----------:|-----:|
| בודקים: "האם ה-DoD שלנו עדיין רלוונטי? חסר משהו? יש משהו מיותר?" | **ב-Retrospective** |
| מוסיפים סעיף שמונע את הבעיה הזו בפעם הבאה | **כשנתקלים בבעיית איכות חוזרת** |
| סעיף שנאכף אוטומטית (למשל ב-CI/CD) — אפשר לשקול אם הוא עדיין צריך להיות ב-DoD הידני | **כשהצוות מגיע לאוטומציה מלאה** |

</div>


### דוגמה: איך DoD מתפתח לאורך 3 ספרינטים

**ספרינט 1 — DoD התחלתי:**

<div dir="rtl">

```
- [ ] קוד נכתב ועובר Compilation
‏- [ ] Unit Tests כתובים ועוברים
- [ ] קוד עבר Code Review
```

</div>


**ספרינט 2 — בעיה: באגים שנמצאו ב-Staging כי לא בדקו שם.**
ב-Retro, הצוות מחליט להוסיף:

<div dir="rtl">

```
- [ ] קוד נכתב ועובר Compilation
‏- [ ] Unit Tests כתובים ועוברים
- [ ] קוד עבר Code Review
+ - [ ] נפרס ונבדק בסביבת Staging          ← חדש!
```

</div>


**ספרינט 3 — בעיה: פיצ'ר "עבד" אבל PO גילה שלא עונה על הדרישות.**
ב-Retro, הצוות מחליט להוסיף:

<div dir="rtl">

```
- [ ] קוד נכתב ועובר Compilation
‏- [ ] Unit Tests כתובים ועוברים
- [ ] קוד עבר Code Review
- [ ] נפרס ונבדק בסביבת Staging
‏+ - [ ] PO אישר בסביבת הבדיקות              ← חדש!
‏+ - [ ] Acceptance Criteria מאומתים          ← חדש!
```

</div>


**התוצאה:** אחרי 3 ספרינטים, ה-DoD כבר חזק יותר. באגים שהיו נמצאים ב-Review נמצאים עכשיו הרבה קודם.

### כללים לעדכון DoD

<div dir="rtl">

| הסבר | כלל |
|------:|-----:|
| באג חזר? פיצ'ר הוחזר? הוסיפו סעיף שמונע את זה | **מוסיפים כשכואב** |
| רק אם הסעיף באמת הפך אוטומטי לחלוטין (למשל — CI/CD אוכף אותו) | **מסירים בזהירות** |
| שינוי DoD הוא החלטה צוותית, לא של אדם בודד | **כל הצוות מסכים** |
| אם יש יותר מ-10 סעיפים ברמת Task — כנראה שיש שם דברים שצריכים להיות בתהליך אוטומטי, לא ברשימת תיוג ידנית | **שמרו על DoD קצר** |

</div>

## 4.4.5 Azure DevOps — Custom Fields, Checklists

### 3 דרכים ליישם DoD ב-Azure DevOps

#### דרך 1: שדה Description ב-Work Item Template (מומלץ להתחלה)

הדרך הפשוטה ביותר — הוסיפו את רשימת ה-DoD לתוך ה-Template של כל Work Item.

**איך:**

1. לכו ל-**Project Settings** → **Boards** → **Team Configuration**
2. בחרו את לשונית **Templates**
3. ערכו את ה-Template של **Task** (או **User Story**)
4. בשדה **Description**, הוסיפו את רשימת ה-DoD כ-Checklist:

<div dir="rtl">

```html
‏<b>Definition of Done:</b>
‏<ul>
‏  <li>[ ] קוד נכתב ועובר Compilation</li>
‏  <li>[ ] Unit Tests כתובים ועוברים</li>
‏  <li>[ ] קוד עבר Code Review</li>
‏  <li>[ ] תיעוד עודכן (אם רלוונטי)</li>
‏</ul>
```

</div>


5. מעכשיו, כל Work Item חדש ייווצר עם רשימת ה-DoD בפנים

> **יתרון:** פשוט, מהיר, לא דורש הגדרות מיוחדות.
> **חיסרון:** לא ניתן לעקוב/לדווח על סעיפים בנפרד.

#### דרך 2: Wiki Page משותף לצוות

צרו עמוד Wiki ב-Azure DevOps עם ה-DoD המלא של הצוות.

**איך:**

1. לכו ל-**Wiki** (בתפריט הצדדי)
2. צרו עמוד חדש בשם **"Definition of Done — הצוות שלנו"**
3. הדביקו את רשימות ה-DoD מסעיף 4.4.2
4. הצמידו (Pin) את הקישור ב-**Dashboard** של הצוות
5. הפנו אליו מכל Work Item Template

<div dir="rtl">

```
‏Azure DevOps → Wiki → New Page → "Definition of Done"
                                      │
                                      ▼
‏                     Dashboard → Add Widget → Markdown
                     → הוסיפו קישור לעמוד ה-Wiki
```

</div>


> **יתרון:** מקום מרכזי אחד, קל לעדכן, כולם רואים את אותו דבר.
> **חיסרון:** צריך לזכור לפתוח את ה-Wiki — לא "מחובר" ל-Work Item.

#### דרך 3: Custom Field מסוג Boolean (מתקדם)

למי שרוצה לעקוב ולדווח — הוסיפו Custom Fields לכל סעיף ב-DoD.

**איך:**

1. לכו ל-**Organization Settings** → **Process**
2. בחרו את ה-Process שלכם (למשל Agile, Scrum)
3. בחרו את ה-Work Item Type (למשל Task)
4. לחצו **+ New field** והוסיפו:
   - **Name**: `DoD - Code Review` | **Type**: Boolean
   - **Name**: `DoD - Unit Tests` | **Type**: Boolean
   - **Name**: `DoD - Deployed to Staging` | **Type**: Boolean
5. חזרו על התהליך לכל סעיף ב-DoD
6. צרו **Query** שמציג Work Items שלא עומדים ב-DoD:

```
Work Item Type = Task
AND State = Resolved
AND (DoD - Code Review = False
     OR DoD - Unit Tests = False
     OR DoD - Deployed to Staging = False)
```

> **יתרון:** ניתן לדוח, לעקוב, לייצר Dashboards על עמידה ב-DoD.
> **חיסרון:** דורש הגדרה ברמת Organization, עבודה נוספת לכל Work Item.

### מה מומלץ?

<div dir="rtl">

| הדרך המומלצת | שלב הצוות |
|-------------:|-----------:|
| דרך 1 (Template) + דרך 2 (Wiki) — פשוט ומהיר | **מתחילים** |
| דרך 2 (Wiki) + דרך 3 (Custom Fields) — מדיד ומעקב | **מתקדמים** |

</div>
