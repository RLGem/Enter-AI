# 5.1 — כרטיסי עזר מהירים (Cheat Sheets)

**חלק:** ה׳ — חומרי עזר מהירים
**גרסה:** 1.0
**תאריך:** 26.02.2026
**קהל יעד:** כל התפקידים — מפתחים, Product Owner, Scrum Master, לקוחות
**סביבת עבודה:** Azure DevOps | ספרינטים של 3 שבועות
**שימוש:** להדפסה על A4, למינציה, ותליה על הקיר

---

> **הוראות הדפסה:** כל כרטיס מתוכנן כעמוד בודד. הדפיסו כל כרטיס בנפרד על A4.
> מומלץ: למינציה + תליה בחדר הצוות / ליד כל עמדת עבודה.

## 5.1.2 — כרטיס Sprint Planning (תכנון ספרינט)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │

│  ┌──────────────┐  ┌──────────────────────────────────────────────────┐     │
‏│  │  Timebox:    │  │  משתתפים:                                       │     │
│  │  3–4 שעות    │  │  PO (חובה) | SM (חובה) | מפתחים (חובה)         │     │
│  │  (ספרינט     │  │  מנהלים / לקוח: לא מוזמנים                      │     │
│  │   3 שבועות)  │  │                                                  │     │
│  └──────────────┘  └──────────────────────────────────────────────────┘     │
│                                                                             │
├──────────────── הכנה לפני הפגישה (Prep Checklist) ───────────────────────── │
│                                                                             │
‏│  Product Owner:                                                             │
‏│  [ ] Backlog מסודר ומתועדף — Stories בסדר עדיפות ב-Azure DevOps            │
‏│  [ ] Stories מוכנים (Definition of Ready) — כולל AC ו-Story Points         │
│                                                                             │
‏│  Scrum Master:                                                              │
‏│  [ ] Capacity מחושב ומעודכן ב-Azure DevOps (חופשות, מילואים, הדרכות)       │
│  [ ] חדר/כלי מוכן — Board פתוח, Velocity של 3 ספרינטים אחרונים מוצג       │
│                                                                             │
│  מפתחים (Developers):                                                       │
│  [ ] סקרו את ה-Stories המועמדים לספרינט — הגיעו עם שאלות מוכנות            │
│  [ ] עדכנו זמינות אישית (חופשות, מילואים) ב-Azure DevOps Capacity          │
│                                                                             │
├──────── חלק 1: מה (What) ──────────┬──── חלק 2: איך (How) ──────────────── │
│                                     │                                       │
‏│  PO מציג Stories לפי עדיפות        │  הצוות מפרק כל Story ל-Tasks         │
│  הצוות שואל שאלות הבהרה            │  הערכת שעות לכל Task                  │
│  הצוות בוחר Stories לפי Capacity   │  זיהוי תלויות בין Tasks              │
│  ניסוח Sprint Goal משותף           │  אימות: סה"כ שעות ≤ Capacity         │
│                                     │                                       │
‏├──────────────── Sprint Goal — תבנית ──────────────────────────────────────── │
│                                                                             │
│  "בספרינט הזה אנחנו מתמקדים ב-________________________                    │
│   כדי לאפשר ל-________________________ לעשות ________________________."    │
│                                                                             │
│  דוגמה: "בספרינט הזה אנחנו מתמקדים בהשלמת מודול ההרשאות                    │
│   כדי לאפשר ללקוח להתחיל UAT על ניהול משתמשים."                             │
│                                                                             │
├──────────────── נוסחת Capacity (קיצור) ──────────────────────────────────── │
│                                                                             │
‏│  Capacity = (ימי עבודה בספרינט) x (שעות ביום) x (מספר מפתחים)              │
│             — חופשות — מילואים — הדרכות — תקורה (20%)                       │
│                                                                             │
│  דוגמה: 15 ימים x 7 שעות x 4 מפתחים = 420 שעות                             │
│          420 — 28 (חופש) — 84 (תקורה 20%) = 308 שעות זמינות                 │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

│  └─────────────────────────────┘  └─────────────────────────────────┘       │
│                                                                             │
├──────────────── כללי Action Items ────────────────────────────────────────── │
│                                                                             │
│  כל Action Item חייב לעמוד ב-4 כללים:                                      │
│                                                                             │
│  1. ספציפי (Specific) — לא "נשתפר בתקשורת" אלא "נכתוב סיכום                │

│                                                                             │

│                                                                             │

## 5.1.6 — כרטיס User Story (סיפור משתמש)

<div dir="rtl">

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
‏│                    USER STORY — כרטיס עזר                                   │
│                                                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  תבנית User Story (User Story Template):                                   │
│                                                                             │
│  ╔═══════════════════════════════════════════════════════════════════╗       │
│  ║                                                                   ║       │
│  ║   בתור   [תפקיד]         ← מי צריך את זה?                        ║       │
│  ║   אני רוצה   [פעולה]     ← מה הוא רוצה לעשות?                    ║       │
│  ║   כדי ש-   [ערך]         ← למה? מה הערך העסקי?                   ║       │
│  ║                                                                   ║       │
‏│  ║   As a [role], I want [action], so that [value].                  ║       │
│  ║                                                                   ║       │
│  ╚═══════════════════════════════════════════════════════════════════╝       │
│                                                                             │
├──────────────── בדיקת INVEST ─────────────────────────────────────────────── │
│                                                                             │
│  כל Story טוב עומד ב-6 קריטריונים:                                         │
│                                                                             │
‏│  I — Independent (עצמאי): אפשר לפתח בלי תלות ב-Stories אחרים              │
‏│  N — Negotiable (ניתן למו"מ): לא חוזה נוקשה — מקום לדיון                   │
‏│  V — Valuable (בעל ערך): מספק ערך למשתמש או לעסק                           │
‏│  E — Estimable (ניתן להערכה): הצוות יכול להעריך את המאמץ                    │
‏│  S — Small (קטן): נכנס בספרינט אחד (מקסימום 8 Story Points)               │
‏│  T — Testable (ניתן לבדיקה): אפשר לכתוב מבחן שיוכיח שזה עובד             │
│                                                                             │
‏├──────────────── Acceptance Criteria — פורמט (Given/When/Then) ────────────── │
│                                                                             │
│  בעברית: בהינתן / כאשר / אז                                                │
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────┐        │
│  │  בהינתן  (Given)  — מצב התחלתי / הנחות מוקדמות                 │        │
│  │  כאשר    (When)   — הפעולה שהמשתמש מבצע                        │        │
│  │  אז      (Then)   — התוצאה הצפויה                               │        │
│  └─────────────────────────────────────────────────────────────────┘        │
│                                                                             │
├──────────────── דוגמה מלאה ───────────────────────────────────────────────── │
│                                                                             │
│  כותרת: התחברות עם שם משתמש וסיסמה                                         │
│                                                                             │
‏│  Story:                                                                     │
│  בתור משתמש רשום,                                                           │
│  אני רוצה להתחבר למערכת עם שם משתמש וסיסמה,                                │
│  כדי שאוכל לגשת לאזור האישי שלי.                                            │
│                                                                             │
‏│  Acceptance Criteria:                                                       │
│                                                                             │
‏│  AC1: בהינתן שם משתמש וסיסמה נכונים,                                       │
│       כאשר המשתמש לוחץ "התחבר",                                             │
│       אז המערכת מעבירה לדף הבית האישי תוך 3 שניות.                          │
│                                                                             │
‏│  AC2: בהינתן סיסמה שגויה,                                                  │
│       כאשר המשתמש לוחץ "התחבר",                                             │
│       אז מוצגת הודעת שגיאה "שם משתמש או סיסמה שגויים"                      │
│       והמשתמש נשאר בדף ההתחברות.                                            │
│                                                                             │
‏│  AC3: בהינתן 5 ניסיונות כושלים רצופים,                                      │
│       כאשר המשתמש מנסה שוב,                                                 │
│       אז החשבון ננעל ל-15 דקות ומוצגת הודעה מתאימה.                          │
│                                                                             │
‏│  Story Points: 5                                                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 5.1.7 — כרטיס Definition of Done (הגדרת "גמור")

<div dir="rtl">

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
‏│                  DEFINITION OF DONE — כרטיס עזר                             │
│                                                                             │
│          "סיימתי" = עומד בכל הסעיפים ברמה הרלוונטית                         │
│                                                                             │
‏├──────────────── DoD ברמת Task ────────────────────────────────────────────── │
│                                                                             │
│  [ ] קוד נכתב ועובר Compilation ללא שגיאות                                 │
‏│  [ ] Unit Tests כתובים ועוברים                                              │
│  [ ] קוד עבר Code Review ואושר ע"י מפתח נוסף אחד לפחות                    │
│  [ ] תיעוד עודכן (אם רלוונטי)                                              │
│                                                                             │
‏├──────────────── DoD ברמת Story ───────────────────────────────────────────── │
│                                                                             │
│  [ ] כל ה-Tasks של ה-Story עומדים ב-Task DoD                               │
‏│  [ ] Acceptance Criteria — כל הקריטריונים מאומתים ועוברים                   │
│  [ ] בדיקות אינטגרציה (Integration Tests) עוברות                           │
│  [ ] הקוד נפרס לסביבת Staging/Dev בהצלחה                                   │
‏│  [ ] PO אישר את הפיצ'ר בסביבת הבדיקות                                     │
│                                                                             │
‏├──────────────── DoD ברמת Sprint ──────────────────────────────────────────── │
│                                                                             │
│  [ ] כל ה-Stories שהוכנסו לספרינט עומדים ב-Story DoD                       │
‏│  [ ] Sprint Review בוצע — הודגמה תוכנה עובדת ללקוח                         │
‏│  [ ] Sprint Retrospective בוצע — Action Items מתועדים                      │
│  [ ] אין Work Items תקועים ב-Active (הכל Done או חזר ל-Backlog)            │
│                                                                             │
‏├──────────────── DoD ברמת Release ─────────────────────────────────────────── │
│                                                                             │
│  [ ] כל ה-Stories עברו QA בסביבת Staging                                   │
│  [ ] בדיקות רגרסיה (Regression Tests) עברו בהצלחה                          │
‏│  [ ] Release Notes מוכנים — רשימת שינויים ברורה                             │
│  [ ] תוכנית Rollback קיימת ומתועדת                                         │
│                                                                             │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
‏│                  Task DoD  ──►  Story DoD  ──►  Sprint DoD  ──►  Release   │
│                                                                             │
│  כל רמה בונה על הרמה שמתחתיה. Sprint לא יכול להיות Done אם ה-Stories      │
│  לא Done. Story לא יכול להיות Done אם ה-Tasks לא Done.                     │
│                                                                             │
│  ╔═══════════════════════════════════════════════════════════════════╗       │
│  ║  כלל ברזל: אם Item לא עומד בכל סעיפי ה-DoD — הוא לא Done.     ║       │
│  ║  לא "כמעט Done". לא "Done חוץ מ...". פשוט לא Done.             ║       │
│  ╚═══════════════════════════════════════════════════════════════════╝       │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

</div>
